Random Module

The function randint() works well for producing random numbers but it has the disadvantage that some of those numbers will repeat themselves. In come choice() and sample().

  • choice(sequence) chooses a random number from a sequence and returns it. The sequence can be a list, for example;
  • sample(sequence, sample) does the same thing, only it returns more numbers, the sample. Somewhat like a lottery pick;

And here is the code:

from random import choice, sample A = [] for i in range(11): A.append(i) print(A) print(choice(A)) print(sample(A, k=5))

Packages

A package is basically a group of modules. And modules seem easy to understand at first. You import time, they do their job: great! You can also pick and choose functions from modules and they behave as expected. All pretty much intuitive.

The __name__ variable

At the same time, there are other things going on in the background. When you import an run, there is some sort of a compiler being built in a __pycache__ folder. Also, there is a special __name__ variable being created. It is used to detect the active context being used.

And here is why we use 'if __name__ == "__main__"
  • when you run a file directly, its __name__ variable is set to __main__;
  • when a file is imported as a module, its __name__ variable is set to the file's name (excluding .py)

So if the name is main, it means the file is run directly.

And if the name is the same as a module, it means the code is run via the module being named.

print("This is some stupid code I need the module for.") if __name__ == "__main__": print("This Python file is run directly.") else: print("This Python file is run through an import command.")
Two Underscores (__variable)

If you want to signal to users that this variable should not be changed, you precede the variable's name with either _ (one underscore) or __ (2 underscores). This does not really make the variable immutable, like a tuple, but makes everyone aware it should not be tampered with.

print("This is some stupid code I need the module for.") __counter = 0 def suml(the_list): global __counter __counter += 1 the_sum = 0 for element in the_list: the_sum += element return the_sum def prodl(the_list): global __counter __counter += 1 prod = 1 for element in the_list: prod *= element return prod if __name__ == "__main__": print("I prefer to be a module, but I can do some tests for you.") my_list = [i+1 for i in range(5)] print(suml(my_list)) print(prodl(my_list))

What still needs to be found out is what the two underscores after the variable do. But most likely it's also just a convention.

__init__.py

Packages may require initialisation, just like modules. Python expects a file with a unique name inside the package's folder: __init__.py. !The content of this file is executed whenever any of the package's modules are imported! This file can be placed in any of the package's subfolders as well.

from sys import path path.append('..\\packages') import extra.good.best.sigma from extra.good.best.tau import funT print(extra.good.best.sigma.funS()) print(funT())

The thing to remember is that you need that __init__ file somewhere and it is better if it has a function with the same name in it. What is still a mystery is not how you invoke different functions from within a package but how do you create that __init__ file.

Two things to remember here:

  1. There is a 'path' variable that stores import directory list.
  2. This 'path' variable is contained in the sys module.
